# Deployment Method Separation Design

**Date:** 2026-03-13
**Status:** Approved
**Scope:** harvester-rke2-cluster (primary) + harvester-rke2-svcs (secondary)

## Problem Statement

The repository mixes two independent deployment methods (Rancher API and Terraform)
at the top level with no clear separation. Additionally:

- The node-labeler operator is deployed by the Terraform method but is unnecessary
  for the Rancher API method (which uses `machineSelectorConfig` for native node labels).
- Operator deployment (cluster-autoscaler, storage-autoscaler, node-labeler) is bundled
  as always-on, but production uses Fleet GitOps for operator lifecycle — making the
  built-in deployment redundant by default.
- The Fleet repo (`harvester-rke2-svcs`) hardcodes cluster-autoscaler scale-down values
  instead of parameterizing them.

## Design

### Directory Structure (Approach B)

```
harvester-rke2-cluster/
├── deploy-api/                          # Rancher API deployment method
│   ├── rancher-api-deploy.sh
│   ├── destroy-cluster.sh               # NEW: wraps --delete
│   └── .env.example
├── deploy-terraform/                    # Terraform deployment method
│   ├── terraform.sh
│   ├── destroy-cluster.sh               # moved from root
│   ├── terraform.tfvars.example
│   ├── *.tf                             # all .tf files from terraform/
│   └── .terraform.lock.hcl
├── operators/                           # STAYS at root (shared by both)
│   ├── images/
│   ├── manifests/
│   │   ├── cluster-autoscaler/
│   │   ├── storage-autoscaler/
│   │   ├── node-labeler/
│   │   ├── cnpg-system/
│   │   ├── mariadb-operator/
│   │   └── redis-operator/
│   ├── templates/
│   ├── upstream/
│   ├── node-labeler/                    # source code
│   ├── storage-autoscaler/              # source code
│   └── push-images.sh
├── prepare.sh                           # STAYS at root (generates config for both)
├── nuke-cluster.sh                      # STAYS at root (method-agnostic)

├── README.md
└── LICENSE
```

| `terraform/*.tf` | `deploy-terraform/*.tf` | All .tf files |
| `terraform/terraform.sh` | `deploy-terraform/terraform.sh` | |

### Terraform State and Provider Cache

- `.terraform/` (root) — provider cache, recreated by `terraform init`. Not moved;
  `terraform.sh` will run `terraform init` from `deploy-terraform/`, creating
  `deploy-terraform/.terraform/` automatically. The old root `.terraform/` can be
  deleted after migration.
- `terraform.tfstate` — the project uses a Kubernetes backend (`terraform.sh` manages
  this). The local `.terraform/terraform.tfstate` is the backend config cache, not
  actual state. It gets recreated by `terraform init`.
- `.terraform.lock.hcl` — commit this file in `deploy-terraform/` (Terraform best
  practice for reproducible provider versions).

### Runtime Config File Locations

Each method's config lives inside its own directory:

- `deploy-api/.env` — generated by `prepare.sh`, read by `rancher-api-deploy.sh`
- `deploy-terraform/terraform.tfvars` — generated by `prepare.sh`, read by terraform

### Script Modifications

#### `deploy-api/rancher-api-deploy.sh`

- Source `.env` from same directory (`deploy-api/.env`)
- Resolve all file paths relative to the **project root** (`REPO_ROOT`), not `SCRIPT_DIR`.
  The script sets `REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"` and credential
  file defaults in `.env.example` use `${REPO_ROOT}/` prefix:
  ```
  PRIVATE_CA_PEM_FILE="${REPO_ROOT}/private-ca.pem"
  CLOUD_PROVIDER_KUBECONFIG_FILE="${REPO_ROOT}/harvester-cloud-provider-kubeconfig"
  CLOUD_CRED_KUBECONFIG_FILE="${REPO_ROOT}/kubeconfig-harvester-cloud-cred.yaml"
  ```
  This keeps credential files at the project root (shared) while scripts live in subdirs.
- Add operator deployment section (opt-in):
  - `DEPLOY_OPERATORS=false` (default OFF)
  - When enabled, deploys: cluster-autoscaler + storage-autoscaler
  - Does NOT deploy node-labeler (`machineSelectorConfig` handles labels)
  - Reads manifests/templates from `../operators/`
  - Creates secrets (cloud-config, ca-cert) for cluster-autoscaler
  - Pushes storage-autoscaler image to Harbor via `../operators/push-images.sh`
- New `.env` variables with defaults:
  ```
  DEPLOY_OPERATORS=false
  CLUSTER_AUTOSCALER_VERSION="v1.34.3"
  AUTOSCALER_SCALE_DOWN_UNNEEDED_TIME="30m0s"
  AUTOSCALER_SCALE_DOWN_DELAY_AFTER_ADD="15m0s"
  AUTOSCALER_SCALE_DOWN_DELAY_AFTER_DELETE="30m0s"
  AUTOSCALER_SCALE_DOWN_UTILIZATION_THRESHOLD="0.5"
  ```

#### `deploy-api/destroy-cluster.sh` (new)

- Simple wrapper calling `rancher-api-deploy.sh --delete`
- Sources `deploy-api/.env` for cluster name display

#### `deploy-terraform/terraform.sh`

- Runs from `deploy-terraform/` — all `terraform` commands execute in this directory
- `terraform.tfvars` is now in the same directory (no path change needed for TF itself)
- `_get_tfvar_value` / `_get_tfvar_heredoc` helpers read from `${SCRIPT_DIR}/terraform.tfvars`
- Update `kubeconfig-harvester.yaml` path reference to `${REPO_ROOT}/kubeconfig-harvester.yaml`
- Update operator directory references to `${REPO_ROOT}/operators/`
- Set `REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"` for cross-directory references

#### `deploy-terraform/*.tf`

- `operators.tf`: Update all paths to `../operators/` (templates, manifests, images, push-images.sh)
- `variables.tf`: Change `deploy_operators` default to `false`
- Keep node-labeler deployment (Terraform provider cannot set node labels)

#### `prepare.sh`

- Write `.env` to `deploy-api/.env`
- Write `terraform.tfvars` to `deploy-terraform/terraform.tfvars`
- Update example file source paths

#### `nuke-cluster.sh`

- Update `.env` source path to `deploy-api/.env` — this script uses Rancher API
  credentials (URL, token, cluster name) which are defined in `.env`. It does not
  need Terraform config. This is correct because `nuke-cluster.sh` performs direct
  Rancher API cleanup regardless of which deployment method was used.

#### `deploy-terraform/destroy-cluster.sh`

- Update to call `terraform.sh` from current directory

### Operator Deployment Logic (deploy-api)

When `DEPLOY_OPERATORS=true`, post-cluster-creation:

1. Extract kubeconfig from Rancher API
2. Deploy cluster-autoscaler:
   - Apply static manifests (namespace, RBAC, PDB, service) from `../operators/manifests/cluster-autoscaler/`
   - Render deployment template via `sed` substitution from `../operators/templates/cluster-autoscaler-deployment.yaml.tftpl`
   - Create secrets (cloud-config with Rancher URL/token, ca-cert with CA bundle)
   - Apply rendered deployment, wait for rollout
3. Deploy storage-autoscaler:
   - Push image to Harbor via `../operators/push-images.sh`
   - Apply static manifests from `../operators/manifests/storage-autoscaler/`
   - Render and apply deployment template
   - Wait for rollout
4. Skip node-labeler (not needed for API method)

### Template Rendering Strategy

The `.tftpl` template files use Terraform variable syntax (`${var.name}`). For the API
method, these are rendered via `sed` substitution — replacing `${var.name}` patterns
with values from `.env`. Both methods reuse the same template files in `operators/templates/`;
no duplication is needed. The `sed` commands map directly:

```bash
sed -e "s|\${var.cluster_autoscaler_version}|${CLUSTER_AUTOSCALER_VERSION}|g" \
    -e "s|\${var.autoscaler_scale_down_unneeded_time}|${AUTOSCALER_SCALE_DOWN_UNNEEDED_TIME}|g" \
    # ... etc
```

### CI Pipeline Updates

Update `.github/workflows/ci.yml`:

- ShellCheck targets: add `deploy-api/*.sh` and `deploy-terraform/*.sh` to file list
- Terraform jobs: set `working-directory: deploy-terraform` for `terraform init`,
  `terraform validate`, `terraform fmt`, and `tflint`
- Checkov: update `directory` to `deploy-terraform` (or run from root with path filter)
- `terraform fmt -check -recursive` from root still works (finds `.tf` anywhere), but
  `terraform validate` and `tflint` require being in the module directory

### Database Operators

Database operators (CloudNativePG, MariaDB Operator, Redis Operator) are **Fleet-only**
in production. They are not deployed by `deploy-api/` even when `DEPLOY_OPERATORS=true`.
The Terraform method retains the ability to deploy them via existing `deploy_cnpg`,
`deploy_mariadb_operator`, and `deploy_redis_operator` toggles (all default to `false`
after this change).

### Fleet Repo Changes (harvester-rke2-svcs)

Parameterize cluster-autoscaler scale-down values (4 files):

| File | Change |
|---|---|
| `.env.rke2-prod` | Add `AUTOSCALER_SCALE_DOWN_DELAY_AFTER_ADD=15m0s`, `AUTOSCALER_SCALE_DOWN_DELAY_AFTER_DELETE=30m0s`, `AUTOSCALER_SCALE_DOWN_UNNEEDED_TIME=30m0s`, `AUTOSCALER_SCALE_DOWN_UTILIZATION_THRESHOLD=0.5` |
| `.env.example` | Same 4 variables with conservative defaults |
| `scripts/lib/env-defaults.sh` | Add 4 vars to `ENVSUBST_VARS` allowlist |
| `00-operators/cluster-autoscaler/manifests/deployment.yaml` | Replace hardcoded `--scale-down-*` args with `${AUTOSCALER_SCALE_DOWN_*}` variables |

Values aligned to conservative defaults (matching this repo):

- `--scale-down-delay-after-add`: 10m → 15m0s
- `--scale-down-delay-after-delete`: 1m → 30m0s
- `--scale-down-unneeded-time`: 10m → 30m0s
- `--scale-down-utilization-threshold`: 0.5 (unchanged)

## Acceptance Criteria

- GIVEN both deployment methods WHEN a user runs `prepare.sh` THEN config files are
  written to the correct method directory
- GIVEN `deploy-api/` WHEN `DEPLOY_OPERATORS=false` (default) THEN no operators are
  deployed; cluster still gets node labels via `machineSelectorConfig`
- GIVEN `deploy-api/` WHEN `DEPLOY_OPERATORS=true` THEN cluster-autoscaler and
  storage-autoscaler are deployed; node-labeler is NOT deployed
- GIVEN `deploy-terraform/` WHEN `deploy_operators=false` (new default) THEN no
  operators are deployed
- GIVEN `deploy-terraform/` WHEN `deploy_operators=true` THEN all three operators
  (including node-labeler) are deployed
- GIVEN the Fleet repo WHEN `render-templates.sh` runs THEN autoscaler scale-down
  values are substituted from `.env` variables
- GIVEN CI pipeline WHEN triggered THEN all checks pass with new file locations
- GIVEN `.gitignore` WHEN files are staged THEN no secrets are committed

## Rollback Plan

All changes are file moves and path updates. `git revert` of the merge commit restores
the previous layout. No data migration or state transformation is involved.

## Migration Path for Existing Users

If `.env` or `terraform.tfvars` exist at the old root locations, `prepare.sh` will
detect them and print a message directing the user to re-run `prepare.sh` (which now
writes to the new locations). Old config files are not deleted automatically — the user
can review and remove them manually.
